# Introduction

Platform24 manufactures T24 which is a medical device software. The device is intended to be used in health care settings to configure an automated triage of patients. The device can be configured with triage flows that ensures that patients with a specific condition, symtom, or combination of conditions and symtoms, are triaged to the right level of care. 

  

This document details the Verification and Validation plan as implemented on Triage24. The document details the activities conducted to produce objective evidence of fulfillment of the device’s intended use and compliance to MDR 2017/745. Each part of the plan is implemented as applicable in each iteration of the device prior to making it available on the market.

# Validation object and scope

Triage24 is defined in detail in [Triage24 Product Description.](https://doktor24.centuri.se/RegNo/D095) The software system is detailed in the [software architecture documentation](http://catalog.dev.platform24.se/docs). 

## Intended use

Triage24 is a configurable IT-system that is used by Healthcare providers to implement medical practice in the triage of patients upon first point of contact. 

  

The device is intended to be configured and adopted to the conditions and needs of a specific healthcare provider. This is achieved either through a service provided by Platform24 or its affiliates or by the healthcare provider. The configuration of the system is made through a web user interface.

  

The configured implementation of the device allows patients to initiate contact, through an associated application (Patient app), with healthcare providers by answering questions about their medical complaint, medical history, and health status. 

  

Based on their answers, the patient is directed to an appropriate level of care, as determined and configured by the health care provider. Such levels are for example: emergency care, physical appointment, digital appointment or self-care. That care can be provided to the patients through a physical meeting or digital meeting with a healthcare provider. Digital appointments can be facilitated through an associated clinician application (Clinic24) and a patient app (white label).

# Verification and validation strategy

Verification and validation is implemented to provide objective evidence of the suitability to use T24 for its intended use. 

  

## Validation

The benefit of T24 is validated through clinical evaluations, risk management, summative usability evaluations and SW testing. Together, these activities form a complete picture of the safety, performance and usability of the device. These activities are planned as part of the lifecycle management of the device and are implemented at defined stages according to this plan.

  

## Verification

Verifications are performed for different purposes in different stages of the design and development of T24. In general the verifications are performed to ensure compliance to requirements. Compliance is verified for the following requirements:

1. Regulatory requirements as defined in MDR 2017/745 Annex I GSPR
    
2. Use requirements as defined in 82304-1:2017
    
3. System requirements as defined in 82304-1:2017 including SW requirements as defined in 62304:2006.
    

# Clinical Evaluation

Clinical evaluation is a product life-cycle process that uses intended use and main functions as basis to draw up objective evidence of the benefits of using T24. The clinical evaluation shall provide evidence of the fulfillment of the General Safety and Performance requirements as assessed in the [GSPR Checklist](https://doktor24.centuri.se:443/RegNo/D144). It shall also consider the need of clinical evidence connected to the hazardous situations as defined in the Triage24 [Risk management Report](https://doktor24.centuri.se:443/RegNo/D152).

  

Clinical evaluation has been performed according to the Clinical Evaluation plan and is reported in the [Clinical Evaluation Report](https://doktor24.centuri.se:443/RegNo/D125).

# Safety and risk acceptance

The safety of T24 is validated through the risk assessment, risk control, residual risk and overall risk acceptability as defined in the Risk [management process](https://doktor24.centuri.se:443/RegNo/D073). A risk management report that concludes the overall residual risk and risk acceptability is mandatory for the validation of the device.

  

The [Triage24 Risk management report](https://doktor24.centuri.se:443/RegNo/D152) is maintained in line with the lifecycle of the device.

# Software testing

Software testing is performed on changes to configuration items to verify implementation of requirements, safety and proper operation. The gitflow process is applied to put software items under configuration control and defines the stages of the different testing activities. A CI/CD pipeline is used to implement the automated testing. Further details on the CI/CD pipeline can be found in the [CI/CD pipeline documentation](https://app.nuclino.com/Platform24/Engineering/CICD-Pipeline-c7ef4f73-7c05-4042-ad3d-b3d21f0316a3).

  

Testing is divided into the following activities:

1. Manual Peer reviews
    
2. Manual feature testing
    
3. Manual Pre release system testing
    
4. Automated Static code analysis
    
5. Automated Unit testing
    
6. Automated Integration testing
    
7. Automated System testing
    

  

Each category of testing has its own detailed plan and reporting defined below.

## Manual Peer reviews

Prior to any feature being included into any shared Platform24 codebranch the changes are peer reviewed by at least another developer in the team responsible for the service in which the feature is developed. All peer reviews are linked to a specific requirement. The peer review fulfils the following three purposes: 

1. The developer has interpreted the requirement correctly.
    
2. The feature has been developed according to the agreed development practices.
    
3. No technical weaknesses are introduced into the system. 
    
4. Spread the knowledge of the implementation and the feature within the development team to increase the maintainability of the systems.
    

  

Criteria for a successful peer review on push to staging is:

1. The software item is identifiable and included in the [Configuration management](https://doktor24.centuri.se/RegNo/D079).
    
2. SW requirements are documented and traceable to system requirements.
    
3. Risk analysis has been performed and documented in the Risk analysis record.
    
4. Architectural dependencies have been defined as appropriate in the [software architecture documentation](http://catalog.dev.platform24.se/docs). 
    
5. Traceability to requirements and tests has been established.
    
6. The appropriate automated tests have been implemented according to this plan.
    

  

Criteria for push to production is: 

- all software requirements have been tested or otherwise verified;
    
- the traceability between software requirements and tests are established;
    
- test results meet the required pass/fail criteria.
    

The peer reviews are linked to specific requirements and performed as merge requests and these are documented, including reviewers, approvers and discussions, in the version control system Gitlab and linked sets of individual and uniquely identifiable changes, called commits, in the code. The peer review history is stored in the version control system and can be audited at any given time by authorized persons.

  

## Manual feature testing

Manual feature testing is performed by the team dedicated QA engineer who is responsible for the software component, the feature testing can be done for either web or mobile applications but in most cases for both. This feature test is typically done in the development environment and focuses on the specific requirement in depth as opposed to the application as a whole. Any new feature should be thoroughly tested as early as possible to find bugs before the implementation begins. Each feature testing activity is linked exactly to a requirement in jira, similarly to how the preceding peer review is linked to the same requirement. The QA engineer responsible for carrying out the feature test uses the requirements jira ticket to record the status of the test case and any potential feedback or defect found. 

  

### Manual feature testing planning and design

Test processes should be well-planned, defined, and documented. So, effective planning entails the creation of a test plan for the feature. The purpose of creating a test plan is to have a documented way to execute consistent, measurable testing on an enterprise wide basis. A well designed test plan will address the following:

  

1. Identified scope (from release cycle or sprint?)
    
2. Create Test Plan from template x (template supports What to test, not to test and how, by who and when)
    
3. Write test cases
    
4. Execute test
    
5. Write protocol, and create test report
    

  

### Manual feature testing execution and output

The test execution phase begins after the completion of the test planning stage and is carried out with the assistance of test cases from testrail, which are prepared during the test planning phase, based on the pre-decided business requirements. If the results of this execution are similar to the expected or desired results, the feature is considered ready to be merged into staging.

  

## Manual system testing

Prior to the release of a changed software component, it undergoes pre release system testing. It is the sole responsibility of the external QA engineers to carry out the pre release testing and record its result. The pre-release tests are recorded in the test case management system testrail ([https://d24.testrail.io](https://d24.testrail.io)).  All  of the pre release system testing is carried out in the staging environment. The focus of this type of tests is to first test the changes introduced in the software component against their actual requirements in a production like environment. The second focus is to carry out manual regression testing of parts of the system that may have been affected by the introduced changes. 

Any defects that are found in this stage will halt the deployment of the software component and necessitate changes into it in order to comply with requirements. When the system has been successfully tested the deployment of the system to production is triggered by merging the versioned release branch into the master branch. This would in turn cause a continuous integration and deployment run, which as described previously will run automated system tests, and if all those pass, trigger the deployment of the software component into the production environment.

  

### Manual system testing planning and design

When formulating the manual system test plan there are many factors that you need to look at,   start by defining the entry and exit criteria for system testing, as well as a time boundary for the testing. The time boundary is important, as no code changes are allowed during  the manual system testing.

  

- Entry criteria: This may include that all software changes have passed unit testing, a build has passed sanity and smoke testing, and the system test suite is prepared.
    
- Exit criteria: This may include that all prioritize test cases have been executed, all defects have been reported, all defects have either been fixed or deferred for future resolution, and the product owner has signed-off.
    

  

Identify the test cases that will be part of the system test suite and also include the test cases for recently-changed code and for defects that were recently found and fixed. Exclude the test cases for known defects that have yet to be resolved. 

  

### Manual system testing execution and output

During the manual system test execution, the QA engineers will prioritize only on the test cases according to critical and frequently used functionalities testing on staging. Additionally, the scope of the testing also depends on the number of features implemented in addition with the existing core functionality. Once the process of testing is completed, the QA engineer prepares a test summary report that will help communicate the activities, processes, methodologies, related to the process of testing, to the other members as well as the stakeholders of the feature. The test summary report is prepared after the completion of the test run, the report consists of all the necessary information related to the process of testing as well as the results delivered by it. Other features of test summary report are:

  

- A summary of found defects for a milestone, test plans or test runs..
    
- The summary of found defects for test cases in coverage matrix.
    
- A summary of found defects for references and their test cases in a coverage matrix.
    

  
  

## Automated static code analysis

  

Static testing is the most basic and earliest form of testing that is used to assert the correctness, from a pure technical perspective, of the implementation of functionality.

  

### Automated code analysis planning and design

The codebase of Triage24 relies heavily on statically typed programming languages that catch basic implementation errors prior to running tests, either at the time the developer tries to compile their program or when running automatic type analysis programs on the code. While this level of test does not verify the degree to which an implemented feature fulfills a given requirement, it does catch a large number of potential errors that are pure technical or implementation errors. Developers developing a given requirement are responsible for implementing their code to be compliant with this type of static analysis and verification of adherence thereof is assured in the continuous integration pipeline. Should the static analysis and tests fail, the developer of a given feature is responsible for ensuring that this is fixed. The inclusion of this purely technical test tool allows individual developers to focus more effort on developing and verifying the code's functional correctness as the static analysis already takes care of the technical correctness.

  

### Automated code analysis execution and output

Static code analysis is performed both by the developer when developing and testing the code locally but primarily it is performed in the build step of the continuous integration pipeline in gitlab. The resulting output of the static code analysis is logged as part of each respective software components pipeline log.

  

## Automated unit testing

The purpose of a unit test is to test an individual feature or functionality or set thereof in isolation. Unit tests are developed as part of the development of individual features and the person developing the feature in question is always responsible for also developing unit tests verifying the correctness of the implementation. 

  

### Automated unit test planning and design

The automated unit tests are developed in parallel with the code of the software unit. Unit tests are developed for all API contracts to simulate expected input (happy pass) as well as reasonably foreseeable incorrect input. The test is held as part of the configuration item itself in GitLab according to the [software architecture documentation](http://catalog.dev.platform24.se/docs).

  

The automated unit test shall contain:

1. Proper operation of child services
    
2. Input stimuli 
    
3. Pass/Fail criteria
    
4. Database interaction mockup
    
5. Service interaction mockup
    

  

The Pass/fail criteria shall contain as applicable:

1. proper event sequence;
    
2. data and control flow;
    
3. *planned resource allocation; 
    
4. fault handling (error definition, isolation, and recovery) ;
    
5. self-diagnostics;
    
6. *memory management and memory overflows; and
    
7. boundary conditions.
    

  

### Automated unit test execution and output

Unit tests are automatically executed in the CI/CD pipeline and are run prior to the building of a deployable artifact. Should any tests in the suite of unit tests fail the continuous integration would stop and produce an error and a notification of the person who introduces the change which caused the test or tests to fail. Each test run is run in the continuous integration system that is closely integrated into the version control system. This allows individual test runs to be directly linked to specific changes of the code in the version control system. As such, the produced test report, that is an artifact of a test run, can be linked to individual code changes and their authors.

  

The automated unit test generates a test record. The test record provides the necessary information to debug any errors and failed tests or to allow review of test history and success rate.

The test report is stored in the pipeline documentation in Gitlab and is traceable to the software item.

  

### Anomalies in automated unit tests

Failed tests generate a record and log of the test that failed. Failed tests initiate software problem resolution. The developer debugs the code to identify the cause of failure and then repeats the push to the CI/CD pipeline.

  

## Automated integration testing

Another integral part of the Triage24 test process is integration testing, i.e. testing that different components working together produce the correct behaviour. In more concrete terms this is defined as multiple software units, here defined as classes of functions, within a service that are tested together with its owned dependencies such as database and web framework and in particular when inputs to the software under tests is its public API.

  

### Automated integration test planning and design

  

As with unit testing, integration tests are primarily automated tests and the developer implementing a particular feature is responsible for creating integration tests that are then included in the automatic test suite that will continuously be run for the software component in question. A key distinction in the process between unit and integration testing is that while unit testing is required to be created for each change, the testing process only Process Test Version 2 Published: 8/16/2021 Page 6 of 10 Document ID: D080 mandates that a developer creates an integration test when appropriate. The decision of when an integration test is appropriate is up to the discretion of the developer. An example of when an integration test is appropriate is when a change is introduced which adds a public facing API or introduces an interaction with the database that is non trivial in complexity. Examples of when it is acceptable to not create an integration test is when a feature requires only changes to business logic that can be fully verified with unit tests or when interactions with external dependencies are very standard (such as a simple select query to a database). 

  

### Automated integration test execution and output

Integration tests are included in the automated tests suites that are automatically run by the continuous integration system each time an individual commit is pushed or in any other way introduced to the shared code base. In the exact same way as with unit tests this creates a test report that is auditable in the continuous integration system and can, through the version control system, be linked both to the individual change and the requirement that mandated the change. The test report can be downloaded from gitlab and can be found linked in each software components pipeline execution in Gitlab. 

  

### Anomalies in automated integration tests

Failed tests generate a record and log of the test that failed. Failed tests initiate software problem resolution. The developer debugs the code to identify the cause of failure and then repeats the push to the CI/CD pipeline.

  

## Automated system testing

These tests are run immediately prior to, and as a prerequisite to, the deployment of a software component with introduced changes into the staging and production environments. The deployment of a software component to the staging environment is done by creating or committing changes into a versioned release branch of the component in question. This causes an automatic continuous integration and deployment to run that, after running automatic static, unit and integration tests, creates a replica of the production environment, deploying the changed component into that replica environment along with all unchanged components. A set of predefined system test cases are then run against that environment in order to ensure that regressions will not be introduced with the deployment of the new functionality. The suite of API test cases that are run create a test report which can be audited for information of exactly what verifications were made prior to the deployment of a changed component. In the event that all API tests are successful the changed software component is deployed into the target environment.

This automated process is set up to ensure that new releases, which typically include many individual changes of a software component, is predictably tested in an as production-like environment as possible prior to its release into a target environment.

  

### Automated system test planning and design

The automated system tests are created by the same team that developed the system. Automated system tests are a set of instructions that executes API calls to test interaction with the software system. The system test is held separately from the code base of the system to be tested. Changes to software system API needs to be analyzed for impact on automated system tests else the test will fail.

  

The basis for planning and design of the automated system tests are the Ref X software architecture documentation.

  

The test shall include:

1. a predefined set of instructions that renders a defined input stimuli as defined in the software architecture documentation.
    
2. the number of simulated simultaneous users
    
3. the allowed run time threshold value 
    
4. threshold levels as applicable
    
5. responses defined in the API specification of software system in the software architecture documentation as expected outcome
    

  

### Automated system test execution and output

On merge requests to staging and production, the CI/CD pipeline initiates execution of automated system tests. The tests will be implemented in a load test environment and the pipeline is halted unless the test passes all set criteria.

  

Test records are generated on each testrun. The records are retrievable through the CI/CD pipeline log and traceable to the software item and the Jira ticket that initiated the code change and push to staging or production. The test record contains details on:

1. The executed tests;
    
2. the test result (pass/fail and cause of failure);
    
3. the version of software tested;
    
4. relevant hardware and software test configurations;
    
5. relevant test tools;
    
6. date tested; and
    
7. the identity of the person responsible for executing the test.
    

  

### Anomalies in automated system tests

Failed tests generate a record and log of the test that failed. Failed tests initiate software problem resolution. The developer debugs the code to identify the cause of failure and then repeats the push to the CI/CD pipeline.

  

## Software problem resolution

Software problem resolution is implemented to correct any anomalies identified during testing. The problem resolution is carried out by the person responsible for the software item being developed. 

  

Logs and records from the failed tests are reviewed to identify the root cause of an anomaly. 

The software item is corrected and a new commit is pushed to the CI/CD pipeline.
