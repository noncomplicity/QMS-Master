Here’s a concise decision write-up for DocTrace’s system architecture.

DocTrace will use “documents as code.” Authors write in Markdown inside a normal folder structure using an IDE or Obsidian, with YAML front-matter carrying compliance metadata. Git is the single system of record for all controlled documents and quality records. The default branch is protected; all changes flow through short-lived branches and PRs/MRs with required reviewers enforced by CODEOWNERS. Approvals on a PR/MR constitute the electronic signature, whose meaning and linkage to specific commits are defined in the Document Control SOP. Tags create immutable snapshots (for example `doc/SOP-DocControl@vX.Y.Z` and periodic `release/qms@YYYY.MM.DD`), and each tagged release bundles the exact Markdown, generated PDFs, a traceability export, and a manifest with checksums into `records/releases/…`. This gives auditors a cryptographically tied snapshot of content, approvals, and evidence.

Authoring remains pleasant and tool-agnostic. Writers never touch schema details directly: they start from Obsidian/IDE templates that prefill front-matter. A strict but friendly validator runs locally (pre-commit) and in CI to enforce the metadata schema, controlled vocabularies, and link integrity. The schema itself lives versioned in `.qms/metadata-schema.json` with a small migration helper so it can evolve without churn. Controlled vocabularies such as doc types, statuses, and approver roles are maintained as simple YAML files and referenced by the validator. Traceability is continuous: requirement, risk, control, and test IDs appear in text and code, and a lightweight scanner builds a live matrix; builds fail if links go missing.

Publishing is downstream and read-only. CI builds a static documentation website from the same Markdown, producing a `site/` folder that can be deployed on GitHub Pages, GitLab Pages, or any static host. For compliance simplicity and deterministic outputs, the default generator is MkDocs (with Material theme), though the pipeline is intentionally neutral and can swap to Hugo, Astro, or Docusaurus with only the install and build steps changed. PDFs for “controlled copies” are produced by a separate, deterministic Pandoc step; links on the website point to these release-bundled PDFs so what readers access always matches the approved tag.

The CI/CD story is identical across GitHub and GitLab. A Makefile exposes stable targets for checks, traceability export, website build, and release bundling. Pipelines call those targets on PRs/MRs and again on tags, ensuring local and CI behavior match. Tool validation for Git hosting, CI runners, the validator scripts, and the renderers is maintained in-repo per ISO 13485/62304 expectations, with simple intended-use tests, golden-file checks, and negative tests that must fail.

This architecture was chosen after considering Astro and Hugo alongside MkDocs. Astro offers the most modern component model and can be adopted later if interactive documentation becomes important. Hugo is the fastest and also a safe alternative. MkDocs is selected as the default because it is lightweight, deterministic, easy to validate, and excels for documentation without adding operational risk. The consequences are that DocTrace keeps authors in familiar tools, enforces quality with automation rather than process friction, and produces audit-ready snapshots on every release while remaining portable between GitHub and GitLab. The main risks are schema drift and accidental breakage of traceability or links; these are mitigated by templates, a clear SOP, pre-commit checks, CI gates, and scripted migrations. Open points, to be resolved in SOPs, are the precise retention policy for release bundles and whether effective-date and status updates are committed to the file at release or recorded solely in the manifest; either approach is acceptable as long as it’s documented and executed consistently.
